2. 네 질문 3개에 대한 답

질문 임베딩 모델을 따로 올리면 추론 엔드포인트를 지정해줘야 하나?

→ 그렇다. URL 하나만 있으면 된다. 백엔드는 EMBED_BASE_URL + EMBEDDING_MODEL_NAME 두 값만 본다. 배선은 두 갈래:

경로 A (단순) liteLLM 뒤에 둠

임베딩 모델을 liteLLM 설정에 등록만. 따로 올릴 앱 없음. EMBED_BASE_URL = liteLLM 그대로.

EMBED_BASE_URL=http://litellm...:4000/v1
EMBEDDING_MODEL_NAME=<liteLLM 등록명>
경로 B (독립) 임베딩 서버를 별 앱으로

그 앱의 /v1 주소를 직접 박음. 단 OpenAI 호환(/v1/embeddings)이어야 함 (TEI·Infinity·vLLM = OK).

EMBED_BASE_URL=http://embed-svc...:8080/v1
EMBEDDING_MODEL_NAME=<그 서버 모델명>
주의 이 repo엔 임베딩 서버 코드가 없다. 어댑터(클라이언트)만 있음. 경로 B를 택하면 임베딩 서빙 앱을 새로 만들어야 한다. 경로 A면 만들 것 없음 — liteLLM 설정만.
질문 gitea엔 필요한 코드 파일만 올려야 함
repo올림절대 금지

빌드 시 프론트에 VITE_API_BASE_URL=https://<백엔드주소> 주입해야 백엔드를 찾는다.

질문 사내 PG·Neo4j로 데이터 재적재 필요

현재 문제: scripts/ingest.py가 fake 고정(가짜 임베더 + 인메모리)이라 PG/Neo4j에 안 들어간다.

해결: build_services(get_settings())로 바꿔 .env 백엔드(liteLLM 임베딩 + postgres + neo4j)를 쓰게 고친다.

선행 조건:
  • 사내 PG에 마이그레이션 적용 (make migrate)
  • PG에 CREATE EXTENSION vector (권한 필요)
  • liteLLM 임베딩 모델 live
  • data/(googledrive·onedrive·slack) → 적재 매니페스트 필요 (현재 없음)

3. 실행 순서 (내가 제안하는 작업)

1
임베딩 경로 결정 (A vs B) A=liteLLM에 임베딩 등록 / B=독립 임베딩 앱 신규 제작. 이거 먼저 정해야 나머지가 갈림.
2
.env 채우기 완료 grep FILL_ME .env 남은 값: liteLLM 모델명, GH 사용자명 등.
3
도달성 점검 백엔드에서 PG·Neo4j·liteLLM /v1/embeddings 연결 확인 스크립트.
4
ingest.py 실 백엔드로 수정 + data/ 매니페스트 생성 그래야 사내 PG/Neo4j에 재적재 가능.
5
gitea repo 분리 + .gitignore 점검 시크릿·대용량 누출 차단. 프론트/백엔드 각각.
6
배포 프론트=앱 배포(VITE_API_BASE_URL 주입) / 백엔드=K8s(deploy/·charts/ 이 토폴로지로 정리).
